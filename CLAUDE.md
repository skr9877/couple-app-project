# CoupleAppProject

LLM의 흔한 코딩 실수를 줄이기 위한 행동 지침입니다. 프로젝트별 지침이 있다면 이와 병합하여 사용하세요.

트레이드오프: 이 지침은 속도보다 신중함에 우선순위를 둡니다. 사소한 작업의 경우 상황에 맞게 판단하세요.

1. 코딩 전 생각하기
   멋대로 추측하지 마세요. 혼란스러운 부분을 숨기지 마세요. 트레이드오프를 드러내세요.

구현하기 전에:

가정한 사항을 명확히 밝히세요. 불확실하다면 질문하세요.

여러 해석이 가능하다면 이를 모두 제시하세요. 혼자서 독단적으로 결정하지 마세요.

더 간단한 접근 방식이 있다면 제안하세요. 필요하다면 사용자 의견에 반론을 제기해도 좋습니다.

불분명한 부분이 있다면 중단하세요. 무엇이 혼란스러운지 명시하고 질문하세요.

2. 단순함 우선 (Simplicity First)
   문제를 해결하는 최소한의 코드만 작성하세요. 추측성 코드는 금지입니다.

요청하지 않은 기능을 추가하지 마세요.

한 번만 사용할 코드에 추상화를 적용하지 마세요.

요청받지 않은 "유연성"이나 "설정 기능"을 넣지 마세요.

발생 불가능한 시나리오에 대한 에러 처리를 하지 마세요.

50줄이면 될 코드를 200줄로 썼다면, 다시 작성하세요.

스스로에게 물어보세요: "시니어 엔지니어가 보기에 너무 복잡하다고 할까?" 그렇다면 단순화하세요.

3. 수술적 수정 (Surgical Changes)
   꼭 필요한 부분만 건드리세요. 자신이 만든 쓰레기만 치우세요.

기존 코드를 수정할 때:

인접한 코드, 주석, 포맷을 "개선"하려 하지 마세요.

고장 나지 않은 것을 리팩토링하지 마세요.

본인의 스타일과 다르더라도 기존 스타일을 따르세요.

관련 없는 죽은 코드(Dead code)를 발견하면 언급만 하세요. 직접 삭제하지 마세요.

수정으로 인해 남겨진 찌꺼기 처리:

본인의 수정으로 인해 사용되지 않게 된 import, 변수, 함수는 제거하세요.

요청받지 않았다면 기존에 이미 존재하던 죽은 코드는 제거하지 마세요.

테스트 방법: 변경된 모든 줄이 사용자의 요청과 직접적으로 연결되어야 합니다.

4. 목표 중심 실행
   성공 기준을 정의하세요. 검증될 때까지 반복하세요.

작업을 검증 가능한 목표로 변환하세요:

"유효성 검사 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"

"버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"

"X 리팩토링" → "작업 전후로 테스트가 통과하는지 확인하기"

단계가 많은 작업은 간략한 계획을 세우세요:

1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
3. [단계] → 검증: [확인 사항]
   성공 기준이 명확하면 독립적으로 반복 수행이 가능합니다. 기준이 모호하면("작동하게 만들기") 계속 질문해야 합니다.

이 지침이 잘 작동하고 있는 신호: Diff(변경 사항)에 불필요한 수정이 줄어들고, 과도한 복잡성으로 인해 코드를 다시 쓰는 일이 적어지며, 실수한 뒤에 수습하는 대신 구현 전에 명확한 질문이 오가는 상태.
